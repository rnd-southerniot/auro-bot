# INA238 Power Monitor

Hardware, driver, and troubleshooting notes for the INA238 chip that
monitors the Pi compute rail (System 1). Complements the driver header
docstring at
[../../ros2_ws/src/navbot_power/navbot_power/ina238_reader.py](../../ros2_ws/src/navbot_power/navbot_power/ina238_reader.py).

For the broader three-battery power architecture and the critical
Pi USB-C isolation finding, see [../power-architecture.md](../power-architecture.md).

## Chip identity

- **Part:** INA238 (bidirectional, 85 V, ±16-bit current / bus voltage
  / power / temperature monitor with I²C).
- **Our breakout:** Adafruit INA238 (STEMMA QT), 15 mΩ shunt on-board.
- **Revision observed in-situ:** `DEVICE_ID` register (0x3F) reads
  `0x2380`. Upper byte `0x23` is the DIEID (fixed for the part family);
  lower byte `0x80` is the silicon revision.
- **I²C address:** `0x40` (default, A0 and A1 jumpers open).
- **Datasheet:** TI SBOSA20C (INA237, Feb 2021, revised 2024).
  INA237 and INA238 are register-identical; SBOSA20C is the authoritative
  datasheet even though INA238 has its own part page.
- **INA237 note:** SBOSA20**A** (older rev) omits the `DEVICE_ID`
  register entirely. If you reach for an older PDF and can't find the
  0x3F register, that's why — use the **C** revision.

## Wiring

- I²C bus 1 on Pi 5 (GPIO 2 SDA, GPIO 3 SCL, `/dev/i2c-1`).
- Breakout pulled up to 3.3 V on the STEMMA QT side (Adafruit board
  provides the pull-ups).
- 5.1 V rail from the Fulree F35J5V20A5L buck runs through the breakout's
  high-side shunt to the Pi 5V GPIO pin. Rail current and voltage are
  measured relative to this shunt.

## Driver

- Location: [../../ros2_ws/src/navbot_power/navbot_power/ina238_reader.py](../../ros2_ws/src/navbot_power/navbot_power/ina238_reader.py)
- Package: `navbot_power`
- Node name: `ina238_reader`
- Publish rate: 2 Hz (chip samples at ~20 Hz internally with AVG=16;
  the node reads one already-averaged value at 2 Hz).
- Publish topics:
  - `/power/ina238/bus_voltage_v` (Float32, volts)
  - `/power/ina238/current_a` (Float32, amperes, signed)
  - `/power/ina238/power_w` (Float32, watts)
  - `/power/ina238/temperature_c` (Float32, °C)
  - `/power/ina238/shunt_voltage_v` (Float32, volts)
  - `/power/ina238/status` (String, JSON — includes `available` flag
    and a human message for operators)
- Launch:

  ```bash
  ros2 launch navbot_power ina238.launch.py
  ```

  Auto-launched as part of `navbot_bringup/base_lidar.launch.py` and
  `navbot_bringup/imu_localization.launch.py`.

## Key driver configuration

- **ADC_CONFIG** (`0x01`) written to `0xFB6A`:
  - MODE = `0xF` (continuous bus + shunt + temperature)
  - VBUSCT = VSHCT = VTCT = `0x5` (1052 µs conversion each)
  - AVG = `0x2` (16 samples)
- **CONFIG** (`0x00`) bit 4 (ADCRANGE) = 1 → ±40.96 mV full-scale shunt
  range, 1.25 µV/LSB. Trades peak current range for 4× better
  resolution on the Pi rail (we expect <5 A steady, not tens).
- **SHUNT_CAL** (`0x02`) computed from shunt resistance (15 mΩ) and
  desired CURRENT_LSB (~15.2 µA/LSB at this ADCRANGE).

## Troubleshooting matrix

### All `/power/ina238/*` values are zero but `status.available` is true

Most common cause: **System 1 battery pack switch is OFF** and the Pi
is on USB-C wall adapter. The buck has no input, no current flows
through the shunt, the chip reads 0 A / 0 W. Voltage may still read
low-but-nonzero from the Pi-side OR-diode path.

This is **not** a driver bug. See [../power-architecture.md](../power-architecture.md)
for the OR-diode mechanism.

Check:

```bash
# On Pi:
i2cdetect -y 1              # confirm 0x40 present
ros2 topic echo /power/ina238/status --once
```

If `available: true` and the System 1 pack is ON, then inspect the
shunt wiring (one common failure: a shunt breakout with cold-solder
joints reads 0 current but non-zero voltage).

### `/power/ina238/*` topics exist but no messages arrive

- Check `i2cdetect -y 1` shows `0x40`. If missing, the breakout isn't
  enumerating — inspect STEMMA QT cable, try a known-good cable.
- Check `journalctl -u ros2` (or wherever you started the launcher)
  for `INA238 init failed` messages.
- If `DEVICE_ID` readback mismatches, confirm the breakout is actually
  INA238 vs INA237 vs INA219 — they have similar pinouts but different
  register maps. Our driver expects `0x23xx` in the upper byte.

### Current reads always zero but bus voltage looks right

`SHUNT_CAL` (register `0x02`) is 0. CURRENT and POWER registers are
**defined** to read 0 when SHUNT_CAL is 0 — this is not a fault, it is
how TI designed the chip to signal "calibration not yet written."
Confirm the driver ran `init()` fully and didn't hit an exception
before writing SHUNT_CAL.

### Readings are extremely noisy

AVG in ADC_CONFIG might be too low. Our driver uses AVG=16 for
~50 ms effective output period; raising to AVG=64 (`0x4`) gives cleaner
but slower data. Tune ADC_CONFIG in
[ina238_reader.py](../../ros2_ws/src/navbot_power/navbot_power/ina238_reader.py)
if needed.

## Known gotcha

The Pi 5 GPIO 5V pin and USB-C input OR together through internal
protection diodes. **"Pi on USB-C wall adapter" does NOT mean "Pi is
isolated from the System 1 battery rail."** On 2026-04-20, 0.94 A was
measured through this exact INA238 shunt while the Pi was on USB-C
wall power. See [../power-architecture.md](../power-architecture.md#important-usb-c-does-not-isolate-pi-from-this-system).
